Over the past few weeks I have been experimenting with Claude Code as a pair programmer to improve my portfolio website. I want to write about what that experience was actually like, not the marketing pitch version, but the real one, including the things that broke, the bugs that required multiple attempts, and the features that turned out better than I expected.

# What is Claude Code?

Claude Code is Anthropic's AI coding tool that runs directly in your terminal. Unlike a chat interface where you paste code back and forth, Claude Code has direct access to your file system. It can read your entire codebase, make edits, run commands, and reason across multiple files at once.

The key difference for me was context. When I describe a problem, Claude Code already knows my project structure, the CSS conventions I use, which libraries are installed, and how components connect to each other. I do not have to re-explain things every time.

# What I Built

## Site-Wide Search

The first major feature was a command palette, the kind that opens with `Ctrl+K` and lets you search across the site instantly. If you have ever used VS Code or Linear, you know the pattern.

This required building a new component from scratch: a backdrop overlay, a modal panel with an input, grouped results, keyboard navigation, and an event system so the header's search button could trigger it. It was a 750-line addition across ten files in a single session.

I later refined it to only index pages and blog posts, which made more sense than surfacing every project or book in search results.

## Project Cards and Modals

Projects on the home page used to be simple cards with a name, description, and a GitHub link. I wanted a way to show more detail without navigating away from the page, so I added project modals. Click a card and a panel slides in with the full project description, tech stack tags, year, and links.

This introduced a new `projectModal.js` component and a matching CSS module. The animation uses Framer Motion with an `AnimatePresence` fade-and-scale, and the modal traps scroll while open.

## Technical Skills Overhaul

The technical skills section went through the most visible transformation. It started as a plain list of text labels and came out with brand icons for each technology, a consistent two-column pill grid inside each category card, and a new Artificial Intelligence category covering ChatGPT, Claude Code, and NotebookLM.

Getting the layout right across breakpoints took more iterations than I expected. At certain viewport widths between 900px and 1130px, the four-column category grid made each card too narrow to fit the inner two-column pill layout, causing horizontal scroll. Same issue appeared in the 570–615px range. Each breakpoint required a targeted media query fix.

## Interests Section

The interests section was the most iterative part of the entire sprint. It went through several rounds of changes:

The section previously had a "Read More" toggle on each card to hide long descriptions. I removed it entirely and showed all text upfront, which immediately surfaced a new problem: card heights were wildly inconsistent because description lengths vary so much. I solved this with a JavaScript equalization function using `ResizeObserver`, which measured card heights and applied `minHeight` to make the grid uniform. This caused an infinite loop. I switched to `window.addEventListener("resize", ...)` combined with `document.fonts.ready` to handle font-timing edge cases.

Later I trimmed each description down to two focused sentences, which made the height variance much smaller and simplified the equalization logic significantly.

Other improvements to the section:
- Each interest icon now sits in a 46px accent-tinted circle background instead of floating bare above the title
- Cards lift on hover with a Framer Motion spring animation and the border transitions to the accent color
- Creator link pills get a subtle background fill on hover in addition to the border color change
- The seven-card grid switched from CSS grid to flex with `justify-content: center`, so the lone last card centers automatically instead of sitting in the bottom-left corner
- Icon and title are now left-aligned, consistent with the description text

## Blog Table of Contents

The blog's table of contents had a persistent scroll bug. Clicking a section heading would either scroll partway and stop, scroll to the wrong place, or do nothing at all depending on which change had most recently been made.

The root cause went through a few iterations to diagnose. The original implementation assigned heading IDs client-side in a `useEffect`, matching heading text content to the table of contents items. This worked for plain headings but failed silently for any heading with inline markdown formatting, because `h.textContent` would not match the raw markdown text with backticks or bold syntax.

The fix I landed on was injecting heading IDs server-side in `page.js` using simple string post-processing, iterating over the table of contents items in order and replacing `<h1>`, `<h2>` tags one at a time with the appropriate `id` attribute. This does not rely on `marked.Renderer`, which is unavailable in the Next.js edge runtime.

The second issue was that the post uses 37 images. With lazy loading enabled, clicking a late section while at the top of the page would calculate the heading's position before those images had loaded and rendered. The scroll target would be off by the total unrendered height of all the images above that heading. Removing `loading="lazy"` from blog post images fixed this. For a long-form reading experience where a reader scrolls the full article anyway, lazy loading is a net negative.

## Mobile Improvements

The welcome section on mobile was stacking the text above the photo, with text left-aligned inside a centered container. I changed it to center everything on mobile, moved the photo to the top of the stack using CSS `order: -1`, and removed a hardcoded `<br />` tag that was creating an awkward mid-sentence line break on small screens. The break is hidden on mobile and preserved on desktop where the text column is narrower.

The resume page on mobile received a few targeted improvements: the four contact links changed from a stacked column to a two-column grid, list items got slightly more line-height and margin for readability on small screens, and the base font size bumped from 14px to 15px.

## Metadata and Performance

A batch of less visible but important changes shipped in late February and early March:

- JSON-LD structured data was added to every page and blog post
- Custom Open Graph images were generated per blog post using Satori and Resvg
- The book list gained a star rating system
- The table of contents gained an active-section tracker that highlights the current heading as you scroll
- Blog post timestamps were added below each post title

# Reflections

The most useful thing about Claude Code is that it holds the entire project in mind. When I described the interests section layout problem, it read the component, the CSS module, the data file, and the equalization logic together before suggesting anything. That is genuinely different from copying code into a chat window.

The least useful thing is that it can be confidently wrong. The `marked.Renderer` approach for injecting heading IDs was introduced with confidence and broke the table of contents completely. Diagnosing that required understanding the edge runtime environment, which Claude Code did not flag upfront.

The workflow that worked best for me: describe the problem clearly, review every file change before it applies, test in the browser, and push back immediately when something regresses. Claude Code is fast and context-aware, but it still requires you to know your codebase and catch its mistakes.

Overall I shipped more improvements in three weeks than I had in the three months prior. Whether that is a good thing probably depends on how you feel about AI tools in software development. For me, on a personal project where the goal is a polished result, it was worth it.
