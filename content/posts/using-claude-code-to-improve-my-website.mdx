Over the past few weeks I have been experimenting with [Claude Code](https://claude.com/product/claude-code) as a pair programmer to improve my portfolio website. I want to write about what that experience was actually like, not the marketing pitch version, but the real one, including the things that broke, the bugs that required multiple attempts, and the features that turned out better than I expected.

# What is Claude Code?

Claude Code is [Anthropic's](https://www.anthropic.com/) AI coding tool that runs directly in your terminal. Unlike a chat interface where you paste code back and forth, it has direct access to your file system and can read your entire codebase, make edits, run commands, and reason across multiple files at once. I used the Visual Studio Code [extension](https://code.claude.com/docs/en/vs-code) for a seamless integration with my editor.

The key difference from a chat interface is persistent context. Claude Code already knows my project structure, the CSS conventions I use, which libraries are installed, and how components connect. I do not have to re-explain things every session.

# What I Built

## Site-Wide Search

![Books Section](/assets/blog/using-claude-code-to-improve-my-website/search.webp)

The first major feature was a command palette, the kind that opens with `Ctrl+K` and lets you search across the site instantly. If you have ever used VS Code or Linear, you know the pattern.

This required building a new component from scratch: a backdrop overlay, a modal panel with an input, grouped results, keyboard navigation, and an event system so the header's search button could trigger it. It was a 750-line addition across ten files in a single session. I later refined it to only index pages and blog posts, as indexing more areas of the website caused several bugs.

## Project Cards and Modals

![Project Card](/assets/blog/using-claude-code-to-improve-my-website/project-card.webp)

Projects on the home page used to be simple cards with a name, description, and a GitHub/website link. I wanted a way to show more detail without navigating away from the page, so I added project modals. Click a card and a panel slides in with the full project description, tech stack tags, year, and links.

This introduced a new `projectModal.js` component and a matching CSS module. The animation uses Framer Motion with an `AnimatePresence` fade-and-scale, and the modal traps scroll while open.

## Technical Skills Overhaul

![Technical Skills Section](/assets/blog/using-claude-code-to-improve-my-website/technical-skills.webp)

The technical skills section went through the most visible transformation. It started as a plain list of text labels and came out with brand icons for each technology, a consistent two-column pill grid inside each category card, and a new Artificial Intelligence category covering ChatGPT, Claude Code, and NotebookLM.

Getting the layout right across breakpoints took more iterations than I expected. At certain viewport widths between 900px and 1130px, the four-column category grid made each card too narrow to fit the inner two-column pill layout, causing horizontal scroll. The same issue appeared in the 570–615px range. Each breakpoint required a targeted media query fix.

## Interests Section

![Interests Section](/assets/blog/using-claude-code-to-improve-my-website/interests.webp)

The interests section was the most iterative part of the sprint. It previously had a "Read More" toggle on each card. I removed it and showed all text upfront, which immediately surfaced a new problem: card heights were wildly inconsistent because description lengths vary so much. I solved this with a JavaScript equalization function using `ResizeObserver`, which caused an infinite loop. I switched to `window.addEventListener("resize", ...)` combined with `document.fonts.ready` to handle font-timing edge cases. Later I trimmed each description to two sentences, which reduced the height variance enough to simplify the whole approach.

Other improvements:

- Each interest icon now sits in a 46px accent-tinted circle background instead of floating bare above the title
- Cards lift on hover with a Framer Motion spring animation and the border transitions to the accent color
- Creator link pills get a subtle background fill on hover in addition to the border color change
- The seven-card grid switched from CSS grid to flex with `justify-content: center`, so the lone last card centers automatically instead of sitting in the bottom-left corner
- Icon and title are now left-aligned, consistent with the description text

## Blog Table of Contents

![Blog Table of Contents](/assets/blog/using-claude-code-to-improve-my-website/toc.webp)

The blog's table of contents had two separate scroll bugs that took multiple sessions to fully resolve.

The first was an ID assignment problem. The original implementation assigned heading IDs client-side in a `useEffect`, matching heading text content to TOC items. This failed silently for headings with inline markdown formatting, because `h.textContent` would not match the raw markdown text. The fix was injecting IDs server-side in `page.js` via simple string post-processing, replacing `<h1>`, `<h2>` tags one at a time in document order. An earlier attempt using `marked.Renderer` broke the TOC entirely because that API is unavailable in the Next.js edge runtime.

The second issue was lazy-loaded images. One of my posts ([Shad UNB 2025](/blog/shad-unb-2025)) has 37 images. Clicking a section near the bottom while at the top of the page calculated the heading's position before those images had loaded, so the scroll target was off by their combined unrendered height. Removing `loading="lazy"` from blog post images fixed it entirely.

## Mobile Improvements

![Mobile Design Changes](/assets/blog/using-claude-code-to-improve-my-website/mobile-design-changes.webp)

The welcome section on mobile was stacking text above the photo with left-aligned text inside a centered container. I centered everything, moved the photo to the top using CSS `order: -1`, and removed a hardcoded `<br />` tag causing an awkward line break on small screens.

The resume page received a few targeted improvements: contact links changed from a stacked column to a two-column grid, list items got more line-height and spacing for readability, and the base font size bumped from 14px to 15px.

## Metadata and Performance

![Books Section](/assets/blog/using-claude-code-to-improve-my-website/books.webp)

A batch of less visible but important changes shipped in late February and early March:

- [JSON-LD](https://json-ld.org/) structured data was added to every page and blog post for SEO
- Custom Open Graph images were generated per blog post using [Satori](https://github.com/vercel/satori) and [Resvg](https://github.com/thx/resvg-js)
- The book list gained a star rating system
- The table of contents gained an active-section tracker that highlights the current heading as you scroll
- Blog post timestamps were added below each post title

# Lessons Learned

## Describe the problem, not the solution

The best results came when I described what was wrong and why, not what code to write. "The TOC scroll lands in the wrong place on long posts with images" produced a real diagnosis. "Add scroll-margin-top to headings" produced a patch that fixed nothing and created new problems.

## Review every change before accepting it

Claude Code moves fast, and that speed works against you if you stop reading the diffs. After a bad change slipped through early in the sprint and broke the TOC entirely, I made a habit of reading every file change before letting it land, regardless of how straightforward it seemed.

## AI assistance compounds existing understanding

Claude Code was most useful where I already understood the domain. CSS layout bugs, component architecture decisions, responsive breakpoints: these are areas where I could evaluate the output quickly and catch errors. It was least useful in unfamiliar territory because I had no way to judge whether a suggestion was correct or just plausible-sounding.

## Bugs introduced by AI tools require real debugging

When something broke, the fix did not come from asking Claude Code to undo it. It came from reading the code, understanding why it was wrong, and describing the real problem. The tool that introduced the bug is not necessarily the right tool for diagnosing it.

## CLAUDE.md is worth writing

Claude Code supports a [`CLAUDE.md`](https://code.claude.com/docs/en/memory) file in the root of your project. It is loaded into context at the start of every session and acts as a persistent brief for the AI. I added one midway through the sprint and the difference was immediate.

Before it, I had to re-explain project-specific constraints every session: the edge runtime limitations, the CSS variable convention for theming, the rule against hardcoding colors. After, those things were already loaded in. The file does not need to be exhaustive. Even a short document covering your tech stack, key conventions, and known gotchas eliminates repeated context-setting and reduces the chance of the AI repeating the same class of mistake.

# Concerns

![Over-reliance](/assets/blog/using-claude-code-to-improve-my-website/skill-atrophy.webp)

## The risk of over-reliance

The productivity gains are real, but it is very easy to stop reading your own code. When changes land quickly and things appear to work, the temptation is to move on rather than understand what was just added. That is how you accumulate code you cannot explain, bugs you cannot diagnose, and architectural decisions you did not make consciously.

I noticed this around mid-sprint. Features were shipping but my mental model of the codebase was drifting. I had to deliberately slow down and re-read recent changes before continuing. The risk compounds on collaborative projects: if you cannot explain a change during a code review, the tool has done more harm than good regardless of whether the feature works.

## Skill stagnation

The problems Claude Code solves well are the same ones that sharpen your skills when you work through them yourself. Debugging a layout issue, tracing a scroll calculation, reasoning about runtime constraints: these are not just tasks to complete, they are situations where understanding deepens. If an AI resolves them before you engage with them, you get the output without the growth.

For a personal project this trade-off is easier to accept. For someone still building foundational skills, it is worth being deliberate about which problems you hand off and which ones you stay with long enough to learn from.

# Reflections

![Anthropic Logo](/assets/blog/using-claude-code-to-improve-my-website/anthropic-logo.webp)

I shipped more improvements in three weeks than in the three months prior. Whether that is a good thing depends on how you feel about AI tools in software development. For me, on a personal project where the goal is a polished result, it was worth it, as long as I stayed engaged with the code rather than just approving changes.

The workflow that held up best: describe the problem clearly, read every diff before accepting it, test in the browser, and push back immediately when something regresses. Claude Code is fast and context-aware, but it still requires you to know your codebase and catch its mistakes.
